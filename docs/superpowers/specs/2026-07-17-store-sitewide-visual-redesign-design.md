# Wristo Store 全站精品化视觉改造设计

## 目标

将 Wristo Store 从常规应用商城升级为“精品表盘设计商店”：以表盘作品为视觉主角，强化 Wristo 青绿色品牌识别，提高商品浏览效率和购买操作的可理解性，并在桌面端与移动端形成一致体验。

本轮只调整前端展示、布局、交互反馈和无障碍细节。现有 API、路由结构、登录、Paddle 支付、激活、购物车状态、多语言机制和业务规则保持不变。

## 视觉方向

采用“编辑型精品商店 + 局部科技舞台”的混合方向：

- 导航、列表、表单和购买流程使用暖白画布、深墨文字与克制青绿色行动色，保证可读性和交易信任感。
- 首页首屏、商品画廊及精选系列允许使用深色或高对比展示舞台，让圆形表盘像设计作品一样被陈列。
- 版式强调清晰网格、编辑式标题、充足但不松散的留白；不使用无意义渐变、夸张阴影或持续漂浮动画。
- 延续现有 `DM Sans` 正文字体和 `Playfair Display` 展示字体，避免引入新字体依赖；展示字体只用于重要标题，不进入按钮、表单或密集数据区域。

## 全局设计系统

在 `src/style.css` 集中维护语义变量，页面和组件优先消费变量而非重复硬编码：

- 色彩：画布、表面、强调表面、正文、次要文字、边线、品牌、品牌深色、品牌浅色、金色评分和危险状态。
- 圆角：控件、卡片、展示容器三档；胶囊只用于标签、价格和短操作。
- 阴影：仅保留轻边界、卡片悬停和浮层三档，取消大面积重阴影。
- 间距：建立页面、区块、卡片和控件的响应式间距变量。
- 控件：统一 Element Plus 主按钮、次按钮、输入框、下拉菜单、消息提示和骨架屏的颜色、圆角、焦点环与禁用状态。
- 动效：统一 160–260ms 的进入、悬停和状态切换节奏；在 `prefers-reduced-motion: reduce` 下关闭非必要位移与过渡。

所有交互控件保留清晰的 `:focus-visible` 状态，触控目标不小于 44px。图标统一以 Solar 线性/双色图标为主；只有 Element Plus 内建交互需要时保留其图标，Material 图标仅在现有语义无法替代时使用。

## 导航与整体框架

`Header.vue` 保留现有数据、路由和菜单命令，只重组视觉层级：

- 桌面端采用紧凑单行主导航。Logo、核心导航、设备选择、语言、购物车和账户区域在同一视觉基线；搜索入口成为明确但克制的高频操作。
- 分类下拉继续使用真实系列数据，缩略图、作品数和跳转行为保持不变。
- 购物车始终有稳定入口；数量为零时不显示徽标，但入口不因状态变化跳动。
- 移动端顶部固定展示 Logo、搜索、购物车和菜单四个入口，菜单展开后再展示分类、账户、语言与次级入口，避免占据过多首屏。
- Header 使用轻微背景模糊和细边线，不使用厚重投影；滚动时保持内容可辨识。

`Layout.vue` 继续承载 Header、Footer 和 FloatingActions。公共页面宽度、横向留白和底部安全区由统一容器变量控制，避免页面自行使用相互冲突的 `100vw`、负边距或溢出修复。

## 首页

首页保持现有接口并继续并行加载系列、热门和新品数据，调整为作品集式叙事顺序：

1. 品牌首屏：左侧为品牌宣言、简短价值说明和一个主 CTA；次要入口保留激活码、合集或 Studio。右侧为深色作品舞台，展示现有 Garmin 手表资产，不增加远程依赖。
2. 搜索入口：作为首屏后的高频动作，支持原有提交与跳转逻辑，补充热门风格提示但不虚构产品属性。
3. 编辑精选/新品：以横向策展区突出少量作品，其余内容可滚动或进入列表，避免一次展示过多信息。
4. 系列：系列卡根据 slug 或稳定索引获得克制主题色，继续展示真实代表作品与应用数量。
5. 创作者/品牌：强化独立设计师氛围，继续使用现有品牌接口和跳转。
6. 热门作品：采用统一商品网格，提供清晰的“查看更多”入口。
7. Premium/Studio：现有推广内容保留，但降为独立结尾板块，不与商品主叙事争夺首屏。

首页每个区块共享同一标题结构：短标签、主标题、辅助说明和可选操作。加载失败时已成功获取的区块仍可显示；单一区块失败不阻断整页。

## 商品卡片

`ProductCard.vue` 保持整卡可键盘进入、Enter/Space 激活和购物车按钮阻止冒泡的现有行为。

视觉结构：

- 圆形表盘图占据卡片主要面积，置于克制的作品舞台中；图片保持圆形裁切和稳定宽高比。
- 信息顺序为标题、简短状态标签、评分/下载量、价格与购物车操作。
- 标题最多两行；指标使用统一图标与等宽数字表现；价格是购买信息中的最高层级。
- 悬停仅产生轻微上移、边线增强和图片缩放；键盘焦点获得等价视觉反馈。
- 移动端保持两列，缩小间距和辅助信息，不把按钮压缩到不可触控尺寸。

标签只从可靠的现有字段推导：

- `Free`：`price <= 0`。
- `New`：存在 `createdAt` 时按稳定期限判断；基础列表没有该字段时不显示。
- `Popular`：按当前列表语境或可靠下载量阈值判断，并保持规则可测试。
- `Purchased/Activated`：沿用现有 entitlement 或购买状态。
- `AMOLED`、`Minimal` 等风格标签仅在后端真实分类或系列数据可用时显示，不从名称猜测。

## 分类、搜索与列表页

`Categories.vue`、`SearchView.vue`、`TopApps.vue` 和复用商品网格的页面共享同一浏览语法：

- 头部包含系列/搜索标题、说明、结果数量与可选主题色。
- 筛选和排序位于紧凑工具栏；桌面端横向排列，移动端折叠为可触控控件。
- 网格断点统一：宽桌面 4–5 列、平板 3 列、移动端 2 列；列宽和间距不会造成横向溢出。
- 加载状态使用与真实卡片比例一致的骨架屏；空状态提供下一步行动；错误状态提供重试或返回入口。
- 系列主题色只作用于头部强调色、标签或局部背景，不改变主按钮语义色。

现有 API 查询参数、分页/加载策略和路由参数不变。若当前页面没有某类筛选数据，本轮不新增后端筛选能力。

## 商品详情页

桌面端使用左右两栏：

- 左侧为作品画廊和缩略图，延续现有不裁切前景图、环境背景、全屏、键盘和触控行为。
- 右侧为 sticky 商品信息与购买面板，依次呈现系列/状态、标题、创作者、评分与下载量、价格、购买状态、主操作和次操作。
- 主操作优先级保持业务语义：可购买时突出购买；购物车是次操作；已购买时突出激活/下载；“Already purchased” 保持为恢复入口；管理员操作不进入普通用户主层级。
- 兼容设备、安装方式、更新时间和购买状态采用清晰的信息组，不与商品描述混为一段。
- 商品描述、安装说明和设备列表沿用现有数据与 HTML 处理方式，仅改善排版。
- 相关推荐与同系列作品使用现有可获得数据；若没有可靠接口，不伪造内容，也不新增后端合同。

移动端改为单栏，图片优先展示；在存在可执行购买/激活操作时显示固定底部操作栏。操作栏适配 `env(safe-area-inset-bottom)`，页面底部预留等高空间，不遮挡安装说明、浮动按钮或 Footer。

## 购物车与购买流程

`CartList.vue`、`CartListPage.vue`、`PurchaseOptions.vue`、`Checkout.vue`、`AlreadyPurchased.vue` 和成功页继承同一交易视觉语言：

- 商品摘要使用小型圆形表盘、标题、状态和价格的稳定网格。
- 折扣、合计、邮箱、支付和错误提示层级明确，主 CTA 在视觉上唯一。
- Paddle overlay/inline checkout、邮箱校验、购买冲突检查、购物车刷新和清空行为完全保留。
- 登录、激活和已有购买入口不改变跳转地址、参数或 sessionStorage 行为。
- 成功、空状态和错误状态使用一致图标、色彩和可执行下一步。

## 多语言与文案

- 现有文案继续通过 `useI18n()` 获取，不把新的用户可见英文硬编码到组件中。
- 优先复用已有 message key。必须新增的文案为全部受支持语言提供回退策略；至少英文源文案完整，未提供人工翻译的语言沿用现有 i18n 回退机制而不是显示 key。
- 商品名、系列名、创作者名和后端描述不在前端改写。
- ARIA 标签与按钮可见文案保持语义一致。

## 数据流和业务边界

- `useProductStore`、`useCartStore`、`useUserStore`、`useLocaleStore` 和现有 composable 是数据源，不复制请求或业务状态到新组件。
- 不修改 `src/api` 的接口地址、DTO、认证头、支付参数或错误码处理。
- 不修改 `src/router/routes.ts` 的现有路径、命名和多语言前缀生成规则。
- 不改变产品详情近期加入的画廊管理、管理员面板、路由复用重载和恢复失败隔离行为。
- 保留 `src/config/axios.ts` 当前工作树中的用户改动，本轮不覆盖或整理该文件。

## 组件边界

优先复用并精炼现有组件，避免为了纯样式拆出大量薄组件。允许新增以下聚焦组件或工具：

- 统一页面区块标题，负责标题、说明和可选操作的布局。
- 商品标签解析工具，纯函数输出可靠标签，便于测试和多处复用。
- 商品网格骨架，保持卡片比例和响应式列数一致。
- 移动端详情操作栏，只接收已由详情页计算好的状态、文案和事件，不自行判断购买业务。

`Header.vue` 当前体积较大，本轮只在与导航视觉和响应式行为直接相关的区域做定向整理，不进行无关的账户/设备业务重构。

## 动效与反馈

- 首页首次进入采用一次性的淡入与轻微上移，区块延迟不超过 60ms 级差。
- 卡片、筛选、下拉和购物车反馈只在用户操作时触发。
- 不使用持续旋转、自动漂浮、光标跟随、重型 WebGL 或新增动画依赖。
- 购物车继续使用现有消息反馈，同时强化按钮状态变化和数量徽标变化。
- 图片和列表加载保持布局稳定，避免内容加载后大幅跳动。

## 无障碍与响应式

- 所有点击卡片、菜单、筛选和购买操作支持键盘，并使用语义元素或正确的 role。
- 焦点不被 sticky Header、移动底栏或弹层遮挡。
- 图片保留有意义的 alt；纯装饰层使用空 alt 或 `aria-hidden`。
- 颜色对比满足常规文本和交互控件的可读要求，不仅依靠颜色表达状态。
- 在 320px–430px 宽度下保持两列卡片且无横向溢出；详情主操作不会被系统安全区遮挡。

## 测试与验收

遵循测试先行：新标签解析、状态推导、结构契约和移动操作栏条件先写失败测试，再做实现。

自动验证：

- 运行新增的商品标签、页面结构和响应式源码契约测试。
- 运行仓库现有完整 Node 测试集。
- 运行 `npm run build`，覆盖 TypeScript、Vue 构建、Vite 产物和 SEO 预渲染。
- 使用 `git diff --check` 检查空白错误。

浏览器验收至少覆盖：

- 桌面：1440px 首页、分类、搜索、商品详情、购物车和购买选项。
- 移动：375px 首页双列卡片、菜单、分类、商品详情底栏、购物车和购买操作。
- 键盘：Header、商品卡、筛选、详情 CTA、购物车按钮。
- 状态：加载、空、错误、免费、普通付费、购物车中、已购买/已激活。
- 动效：正常模式与 `prefers-reduced-motion`。

如果本地 API 或支付环境无法提供某个实时状态，必须将代码/构建验证与环境限制分开报告，不把未验证状态声明为完成。

## 交付内容

- 首页、商品卡片、分类/搜索列表、商品详情、Header、移动导航、购物车与核心购买页面形成统一视觉系统。
- 汇报主要修改文件、完成的视觉与交互优化、未完成或因环境受限的事项、测试和构建结果。
- 不包含后端接口、路由、支付、登录或核心业务规则变更。
